# Lessons

- 2026-04-01: 审查 PR 时要把“升级内容是否有 breaking change”和“执行 gh pr merge 属于不可逆操作”分开表述，不能都说成“破坏性”。
- 2026-04-01: PR 在远端合并后，如果后续要部署合并结果，先同步远端分支；不要为了部署临时手改本地工作区文件。
- 2026-04-01: 在逐个审 PR 的流程里，用户说“继续”默认表示按前一个节奏继续推进；如果上一个 PR 已给出“可直接合并”，就先合并当前 PR 再看下一个，不能擅自当成“跳过合并”。
- 2026-04-01: 合并 compose 相关 PR 后，要立刻执行对应环境的 deploy；不能只停在 GitHub 合并完成。
- 2026-04-01: 涉及敏感数据时，绝不能直接读取或输出密钥值；只允许按仓库规则查看结构，排查也不能越线。
- 2026-04-03: 排障汇报不能只停在表象，必须把“直接卡住的点”和“根本原因”分开写清楚；这次 `volsync-src-cli-proxy-api` 表象是临时卷挂不上，根因是 VolSync 反复引用了 Longhorn 底层已不存在的旧快照。
- 2026-04-05: 排查网络与 DNS 问题时，必须先逐项核对“宿主机名、虚拟机名、VIP、实际 IP、DNS 记录来源”的对应关系；没核实前不能把 `sakamoto.lan`、`sakamoto-k8s`、`k8s-vip.lan` 当成同一对象来叙述。
- 2026-04-07: 排查 `router-mine` 时默认使用 `root@192.168.6.1` 或 `ssh router-mine`，不能沿用当前工作站用户名直接 SSH 到路由器。
- 2026-04-08: 做迁移类需求时必须严格锁定原始目标和现状边界；如果用户要求“迁移后保持现状”，就不能擅自扩展到“顺便启用新记录/新入口”，并且确认步骤要给出可直接执行的逐条操作。
- 2026-04-09: 做域名托管迁移时必须分别核对 cert-manager 的 DNS01 和 external-dns 的 provider ownership；只迁证书链路不算完成，除非用户明确只要求证书。
- 2026-04-09: 对现有应用做 second-domain 联通测试时，优先沿用 HelmRelease 已有的 route 配置做临时 hostname 调整；不要改走仓库里另放的独立 DNSEndpoint/HTTPRoute 草稿文件，除非用户明确要求。
- 2026-04-09: 向用户汇报结论时禁止使用“不是……而是……”这类对举句式；直接写结论、证据和处理动作。
- 2026-04-09: 排查 Home Assistant 多网卡/zeroconf 问题时，若用户提供 HA 后台“网络适配器”运行时配置，应以该配置为准；不能仅凭 Pod 默认路由或默认接口推断实际监听网卡。
- 2026-04-09: 回复用户纠错时，禁止评价用户的说法或做法是否正确；只记录约束、修正结论，并继续处理问题。
- 2026-04-09: 回复用户时禁止道歉和无关废话；直接写必要事实、结论和动作。
